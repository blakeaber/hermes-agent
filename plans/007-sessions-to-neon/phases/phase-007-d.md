# Phase 007-D: Raw audit — inbound + outbound + tool-call hooks → raw_events

## Goal
Add best-effort hooks at three call sites (Slack inbound, Slack outbound, tool execution) that write to the Neon `raw_events` table for compliance-grade auditability, so every message and every tool invocation is durably recorded with full payload + provenance.

## Context
Plan 007-C made `messages` writes go to Neon. But `messages` is the agent's CONVERSATION view — cleaned-up, single row per user/assistant turn. Compliance + debugging both need a RAW view: every Slack event as received, every tool call with full arguments + return, separate from the agent's view. `raw_events` (Phase 007-A) is the table; this phase wires the writes.

## Dependencies
Phase 007-C complete (SessionStore routes through StorageBackend in saas mode; NeonBackend.append_raw_event exists from Phase 007-B).

## Scope

### Files to Create
- `tests/test_raw_events_audit.py` — ≥3 tests: `test_inbound_event_writes_raw_row`, `test_outbound_event_writes_raw_row`, `test_tool_call_writes_request_and_response_rows`

### Files to Modify
- `gateway/platforms/slack.py` — two sites:
  - `_handle_slack_message` (after dedup, before agent dispatch): write `event_kind="slack_inbound"` with full `event` dict
  - `send` method, around line 840 where `register_output` already fires: write parallel `event_kind="slack_outbound"` with full payload
- `agent/conversation_loop.py` OR `agent/tool_executor.py` (whichever owns tool invocation): two writes per tool call:
  - Before invocation: `event_kind="tool_call_request"` with `{tool_name, arguments, session_id}`
  - After invocation: `event_kind="tool_call_response"` with `{tool_name, result, error, latency_ms, session_id}`

### Explicitly Out of Scope
- Sampling / throttling raw_events writes (write everything; retention is a later policy decision)
- Encryption beyond Neon's at-rest default
- Schema for derived analytics (rollups, aggregates) — query raw at first; aggregate later if needed
- Browser UI for raw_events — separate plan
- Raw_events writes for non-Slack adapters (Telegram/Discord/etc) — out of scope for this plan; pattern is repeatable

## Implementation Notes

1. **Best-effort everywhere.** Every `append_raw_event` call is wrapped in `try/except Exception as e: logger.debug("raw_events write failed: %s", e)`. Audit failures NEVER block the user-facing path. The Plan 004-A `register_output` pattern at `gateway/platforms/slack.py:840` is the reference.
2. **Tenant resolution via cache.** Reuse `self._tenant_id_by_team[team_id]` (Plan 004-A cache). If missing, skip — same gate as register_output uses.
3. **Pool guard.** Use the existing `self._get_neon_pool()` helper at `gateway/platforms/slack.py:1331`. If pool is None (not saas mode), skip silently.
4. **`platform_message_id` for idempotency.** Slack inbound: use `event['ts']`. Slack outbound: use `sent_ts` from the send result. Tool calls: use a deterministic hash of `(session_id, turn_index, tool_name, args)` — generated by the agent loop, passed in.
5. **Tool-call hook placement.** Find the existing point where `agent.tool_executor` dispatches an MCP/tool call and receives the result. Most likely in `agent/conversation_loop.py` (the conversation loop iterates through tool calls). If the tool executor is the right wrapper, hook there. Either way, the wrap should produce request + response paired rows.
6. **Payload sanitization.** Raw payload may contain secrets (API keys, OAuth tokens) if tool arguments include them. Reuse Hermes's existing redactor (the one referenced by the "Secret redaction: ENABLED" startup line) to scrub payload before writing. Same redactor; same boundary.
